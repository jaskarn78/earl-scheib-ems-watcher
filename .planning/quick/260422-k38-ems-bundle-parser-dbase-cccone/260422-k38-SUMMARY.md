# Quick Task 260422-k38: EMS dBase Bundle Parser Summary

CCC ONE EMS 2.01 dBase bundle parser wired into the existing Earl Scheib
watcher — Marco's new CCC ONE writes `.AD1/.VEH/.ENV` dBase files instead of
plain BMS XML, and the watcher now detects those bundles, parses them with
pure-Go `Valentin-Kaiser/go-dbase`, synthesizes BMS 2.6 XML that `app.py`'s
`parse_bms()` already accepts, and POSTs with `?trigger=ems_bundle` so the
server can tell dBase-derived estimates from plain XML ones.

## What Shipped

- **Bundle detection** in the existing 5-minute scan cycle. Any folder
  containing `{GUID}.AD1` + `{GUID}.VEH` (+optional `.ENV/.LIN/...`) becomes
  one bundle. Case-insensitive extension matching.
- **BMS XML synthesis** using stdlib `encoding/xml` in the CIECA namespace
  `http://www.cieca.com/BMS`, emitting every element `app.py parse_bms()`
  looks up: `DocumentVerCode`, `DocumentStatus`, `Owner/GivenName`,
  `Owner/OtherOrSurName`, `Owner/CommPhone`, `EventInfo/RepairEvent/CloseDateTime`,
  `ActualPickupDateTime`.
- **Bundle-aware dedup**: virtual path `<watchFolder>/<basename>.bundle` keyed
  by latest-mtime-anchor across component files. Guaranteed not to collide
  with plain-file dedup rows. Dedup hash = SHA-256 over concatenated file
  bytes sorted by lowercase basename — deterministic across scans.
- **Pipeline coexistence**: bundle track runs AFTER the existing plain-file
  track. Same DB, same Sender. A folder with both a plain `.xml` and a
  complete bundle POSTs both — neither suppresses the other.
- **Signed Windows installer** rebuilt and pushed — zip at repo root
  MD5-matches the one served from `https://support.jjagpal.me/earlscheibconcord/download`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] go mod tidy prunes unused dependency**
- **Found during:** Task 1
- **Issue:** Plan specifies `go get` → `go mod tidy` → cross-compile. After `go get`, `go mod tidy` dropped `github.com/Valentin-Kaiser/go-dbase` because no source file imported it yet. This would leave Task 1's commit without the dep entry. Running cross-compile + verify still worked because `go build ./...` doesn't need unused deps, but the intended Task 1 → Task 2 hand-off relied on the dep being listed.
- **Fix:** Ran `go get` again after tidy to re-add as `// indirect`, committed that state for Task 1. After Task 2's `internal/ems/parse.go` imported the package, `go mod tidy` naturally promoted it to direct require — captured in Task 2's commit.
- **Files modified:** `go.mod`, `go.sum`
- **Commits:** `83b865f` (Task 1), `a4ac1a2` (Task 2)

**2. [Rule 3 - Blocking] go-dbase NewTable requires `.DBF` extension and uppercases full path**
- **Found during:** Task 2
- **Issue:** `dbase.NewTable` in `io_unix.go` `Create()` does `strings.ToUpper` on the entire filename path before opening the file (lines 150, 165). On Linux `/tmp/...` becomes `/TMP/...` which doesn't exist. Also enforces `.DBF` extension at parse time. First-fix attempt (write to `.DBF` in the tempdir then rename) failed because the uppercase-full-path behaviour corrupted the directory component.
- **Fix:** Test helper `makeTestDBF` now creates an uppercase-safe workdir under `t.TempDir()`, calls `t.Chdir` into it, passes a bare `"FIXTURE.DBF"` filename (no path prefix) so the library's ToUpper only affects the filename, not the directory. After close, reads the written bytes and writes them to the caller's requested final path. Tests drop `t.Parallel()` because `t.Chdir` is incompatible with parallel tests (enforced by Go 1.25 testing).
- **Files modified:** `internal/ems/parse_test.go`, `internal/scanner/ems_bundle_test.go`
- **Commits:** `a4ac1a2` (Task 2), `16db1d6` (Task 3)

**3. [Rule 2 - Missing critical functionality] go-dbase NewTable requires Converter**
- **Found during:** Task 2
- **Issue:** First run of the ParseBundle tests failed with `NewTable: no converter specified`. Plan didn't mention the `Converter` field in `dbase.Config`.
- **Fix:** Added `Converter: dbase.NewDefaultConverter(charmap.Windows1252)` to the test fixture config (and implicitly in production via `InterpretCodePage`-fallback when the file has a codepage mark). Pulled in `golang.org/x/text/encoding/charmap` — already a transitive dep from go-dbase, so no new top-level requirement beyond what `go mod tidy` already surfaced.
- **Files modified:** `internal/ems/parse_test.go`, `internal/scanner/ems_bundle_test.go`
- **Commits:** `a4ac1a2`, `16db1d6`

**4. [Rule 3 - Blocking] scanner Run() early-return skipped bundle track**
- **Found during:** Task 3
- **Issue:** Existing `scanner.Run` returns `(0, 0)` early when `Candidates(...)` returns zero files. A watch folder holding ONLY an EMS dBase bundle (no plain `.xml`) would never reach the new bundle track — Marco's real use case breaks.
- **Fix:** Removed the early return but kept the `"no files"` run-note semantics when BOTH tracks are empty (preserving `TestRunEmptyFolderStillRecordsRun`). Otherwise the note is `"scan of <dir> (N xml + M bundle candidates)"`.
- **Files modified:** `internal/scanner/scan.go`
- **Commit:** `16db1d6`
