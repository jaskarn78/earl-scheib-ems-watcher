# Phase 5 — Queue Admin UI: Design Contract

> Visual and interaction contract for the `earlscheib.exe --admin` local browser SPA.
> Generated by gsd-ui-researcher. Consumed by gsd-planner and gsd-executor.

---

## 1. Aesthetic Direction (LOCKED)

**Direction: Industrial Editorial — "Grease & Type"**

The Queue Admin UI reads like a foreman's manifest pinned to a shop wall — not a dashboard, not a SaaS product. The dominant surface is deep graphite (near-black), with warm off-white type printed on it like ink on oil-stained paper. A single aggressive red accent marks the one action that matters: cancel. The layout is asymmetric: a heavy left column anchors a type-led header against a wide right field of job rows with a strong vertical rhythm. There are no rounded cards. There are rules — thick horizontal dividers like ledger lines separating each customer block.

**Why this fits Marco + Earl Scheib:**
- Earl Scheib is an auto body shop. Manual labor, metal, paint. The aesthetic should feel earned and grounded, not tech-startup.
- Marco opens this once a day to glance and cancel. It must read instantly at a glance — like a physical work order sheet, not an app.
- The existing `app.py` support page already uses `#CC0000` red, giving a brand anchor. This UI deepens into dark-mode territory where that red reads as urgent, not decorative.
- CLAUDE.md explicitly prohibits Inter, generic SaaS blues, and pastel palettes — this direction obeys all three prohibitions.

**The one thing Marco will remember:** It looks like the CCC ONE software they use every day — functional, purposeful, slightly intimidating — not like an iPhone app.

## 2. Typography

### Font Pairing

| Role | Font | Source |
|------|------|--------|
| Display / headings | **Bebas Neue** | Google Fonts |
| Body / data / labels | **JetBrains Mono** | Google Fonts |

**Rationale:**
- Bebas Neue is condensed, all-caps, brutally legible at large sizes. It carries the weight of a stencil stamp on sheet metal. It sets the "foreman's manifest" tone before any content is read.
- JetBrains Mono is a monospace body font — unusual for a data UI, intentional here. Phone numbers, timestamps, and repair refs are all alphanumeric codes. Monospace makes them scannable in columns even without a formal table, gives a technical precision feel, and avoids the "Inter + any display font" cliché the project explicitly bans.
- This pairing has never appeared in any GSD project before.

### Type Scale — EXACTLY 4 SIZES

| Role | Size | Weight | Line Height | Font | Notes |
|------|------|--------|-------------|------|-------|
| Page title (h1) AND empty/error state headings | **48px** | 400 (optically bold) | 1.0 | Bebas Neue | `QUEUE`, `QUEUE CLEAR`, `CANNOT REACH SERVER` — same size, context differentiates |
| Customer group heading AND pending count badge in header | **22px** | 400 | 1.1 | Bebas Neue | Customer name (h2) and `{N} PENDING` count reuse this size |
| Body / message preview | **14px** | 400 | 1.6 | JetBrains Mono | Job body preview text, inline cancel failure message |
| Data fields, badge labels, captions, metadata | **12px** | 400 | 1.4 | JetBrains Mono | Phone, time, repair ref, `PENDING` badge label, `Updated {N}s ago`, group message count, `REF:` label |

**Removed sizes (do not use):** 36px, 20px, 13px, 11px. These are eliminated entirely. All content formerly at those sizes maps to one of the four sizes above:
- 36px empty/error headings → 48px (reuses page title)
- 20px pending count in header → 22px (reuses customer group heading)
- 13px body text → 14px
- 11px captions, badge labels, button labels → 12px

**Button label font:** `CANCEL`, `CONFIRM?`, `FAILED — RETRY?` — Bebas Neue is inappropriate for small button text; use JetBrains Mono **12px**, weight 700, uppercase, `letter-spacing: 0.10em`. This is a deliberate exception to the display-font-for-headings rule: button labels are functional controls, not headings.

**Loading:** Both fonts loaded via Google Fonts CDN with `display=swap`. Single `<link>` in `<head>`. No self-hosting required for a local-only tool.

## 3. Color Palette

**Mode: Dark only.** No light mode. Rationale: Marco's shop PC likely has varying ambient light; dark mode is easier to scan in bright-garage or dim-office conditions. Dark mode also makes the red accent pop with maximum urgency. Adding a light-mode toggle adds complexity with no user benefit — this is a single-purpose internal tool, not a product.

| Role | Hex | CSS Variable | Usage |
|------|-----|--------------|-------|
| Dominant (60%) | `#111210` | `--bg` | Page background — near-black with a warm graphite undertone, not pure black |
| Secondary (30%) | `#1C1B18` | `--surface` | Customer group blocks, the "sheet" each customer's jobs sit on |
| Tertiary | `#252420` | `--surface-hover` | Row hover state |
| Accent (10%) | `#CC2200` | `--red` | Cancel button (armed + hover). Error state heading. Nothing else. |
| Accent muted | `#3A1A14` | `--red-dim` | Cancel button background at rest. PENDING badge background. |
| Text primary | `#E8E4DC` | `--text` | Body copy, customer names, PENDING badge label — warm off-white, not pure white |
| Text secondary | `#8E8883` | `--text-muted` | Labels, captions, phone numbers, timestamps (lightened from #7A7570 for contrast) |
| Divider | `#2E2C28` | `--rule` | Horizontal rules between customer groups |
| Border | `#333028` | `--border` | Thin borders on surface blocks |
| Success flash | `#1A3A1A` | `--success-bg` | Transient background on successful cancel |

**Background treatment:** NOT a solid flat color. The `--bg` surface has a subtle grain overlay — a 150px×150px SVG noise pattern tiled at 3% opacity over the background. This adds depth without adding any assets (the SVG is a 12-line inline data URI). No gradient mesh. The grain is the texture, aligned with the "oil-stained paper" metaphor.

**Contrast ratios (WCAG AA):**
- `--text` (#E8E4DC) on `--bg` (#111210): 14.2:1 — AAA
- `--text-muted` (#8E8883) on `--bg` (#111210): **5.4:1** — AA (was #7A7570 at ~4.1:1, which failed; now passing)
- `--text` on `--surface` (#1C1B18): 12.8:1 — AAA
- `--text` (#E8E4DC) on `--red-dim` (#3A1A14) for PENDING badge label: **12.4:1** — AAA (badge text is off-white, not red-on-dark-red)
- White (#FFF) on `--red` (#CC2200) for cancel button text: 5.5:1 — AA
- `--red` (#CC2200) on `--bg` (#111210) for error heading at 48px: 3.4:1 — passes AA for large text (48px Bebas Neue qualifies as large text under WCAG)

**PENDING badge color correction note:** The previous spec declared `--text-red` (#CC2200) as badge text on `--red-dim` (#3A1A14), yielding ~2.8:1, which fails WCAG AA at any size. The badge text is now `--text` (#E8E4DC) on `--red-dim` (#3A1A14). The badge's dark-red fill already signals urgency; off-white label text is more legible and equally on-brand.

**Inline cancel-failure message color:** The error sentence `Could not cancel. Server error — try again.` renders in `--text` (#E8E4DC) at 14px, not in `--red`. The button label `FAILED — RETRY?` and the armed row's red tint already communicate error state. Using `--red` for the sentence below would fail AA at 12px (3.4:1 on #111210) and is unnecessary for comprehension.

## 4. Layout

### Grid

No formal CSS grid framework. Single-column layout, max-width 860px, centered, with 48px horizontal padding on each side (32px on mobile). This is narrow intentionally — a manifest, not a dashboard. Wide layouts invite dashboardy thinking.

```
Page
└── Header bar (full width, 72px tall)
    ├── Left: "QUEUE" title (Bebas Neue 48px) + "Earl Scheib Auto Body" caption
    └── Right: pending count badge (Bebas Neue 22px, --red) + last-updated timestamp
└── Content area (max-width 860px, centered)
    ├── Customer Group Block (repeats)
    │   ├── Customer header: NAME (Bebas Neue 22px) + PHONE (JetBrains Mono 12px)
    │   │   3px top border in --rule. Right-aligned phone. Below: "2 messages pending" (12px --text-muted)
    │   └── Job Row (repeats within customer)
    │       ├── Left: job-type badge (12px JetBrains Mono 700 uppercase) + message body preview (14px)
    │       ├── Center: send-at time (12px) + repair ref (12px)
    │       └── Right: Cancel button
    └── Empty state (full content area)
```

### Queue Grouping: By Customer (LOCKED)

Rationale: Marco's mental model is "did I accidentally schedule a text to the wrong customer?" — not "what's sending first?" Grouping by customer (most-recently-created first) lets him find a customer by name and see all their pending messages in one glance. Time-ordered would require scanning multiple positions for a single customer's messages. "Customer-first" also mirrors how the shop thinks about work orders.

Sort order: customer groups ordered by `MAX(created_at) DESC` (most recently added customer first). Within a customer group, jobs ordered by `send_at ASC` (soonest first).

### Job Row Rendering: Manifest List (not cards)

Each job is a row separated by a thin 1px `--rule` line. No rounded cards, no box shadows. The rows read like line items on a purchase order. Height: minimum 64px per row with 16px vertical padding. Column layout within each row is a CSS flexbox with three flex children: left (type + preview), center (time + ref), right (cancel).

### Page Header / Branding

The header is a full-bleed dark bar (same `--bg` color, with a 1px bottom border in `--rule`). Left side: "QUEUE" in Bebas Neue 48px, tracked slightly wide (`letter-spacing: 0.04em`), followed immediately on the same baseline by a 12px `--text-muted` caption "Earl Scheib Auto Body · Message Queue". Right side: a pending count like "4 PENDING" in Bebas Neue **22px** with the `--red` color, plus a `--text-muted` caption "Updated 12s ago" in JetBrains Mono 12px. The header does NOT scroll away — it is `position: sticky; top: 0`.

### Customer Group Header

3px top border in `--rule` color (decorative ledger line). Customer name in Bebas Neue 22px (`--text`). Phone number in JetBrains Mono 12px (`--text-muted`) on the same line, right-aligned. Below the name, a subtle count: "2 messages pending" in 12px `--text-muted`.

### Empty State

Full content area. Centered vertically and horizontally. No illustration, no icon — just type.

```
QUEUE CLEAR
Nothing scheduled — texts will go out
as new estimates and closed jobs come in.
```

"QUEUE CLEAR" in Bebas Neue **48px** `--text-muted`. Body in JetBrains Mono 14px `--text-muted`, line-height 1.6.

### Error State (server unreachable)

Same centered layout.

```
CANNOT REACH SERVER
The queue server is not responding.
Check that app.py is running, then refresh this page.
```

"CANNOT REACH SERVER" in Bebas Neue **48px** `--red`. Body in JetBrains Mono 14px `--text-muted`.

### Density Decision: Industrial-Mid

Not SaaS-dense (rows are not crammed), not magazine-generous (no excessive white space). Each job row is 64px minimum. Customer group headers are separated from the previous group by 24px of space + the 3px rule. This density is intentional: Marco can see 4–6 pending jobs without scrolling on a standard 1080p laptop.

## 5. Motion & Interaction

### Page-Load Reveal (LOCKED)

One orchestrated stagger on page load only. No per-scroll animations.

1. Header: fades in from `opacity: 0` → `1`, `translateY(-8px)` → `translateY(0)`. Duration: 200ms. Easing: `cubic-bezier(0.22, 1, 0.36, 1)` (ease-out-quint).
2. Customer groups: each group fades in with `opacity: 0` → `1`, `translateY(12px)` → `translateY(0)`. Stagger: 60ms per group. Duration: 240ms each. Same easing.
3. Total reveal: for 5 groups = 200ms header + (5 × 60ms) stagger + 240ms last group = ~740ms total. Snappy, not theatrical.

Implementation: CSS `@keyframes fadeUp` applied with inline `animation-delay` set via JS during render. No external animation library required.

### Hover States

- **Job row hover:** `background-color` transitions to `--surface-hover` (#252420). Duration: 80ms. No transform. No shadow. The row "warms" slightly.
- **Cancel button hover:** `background-color` transitions from `--red-dim` to `--red`. The button "ignites." Duration: 100ms. No transform (no lift — this is not a playful action).
- **Customer group hover:** No hover effect on the group header itself — only individual job rows respond.

### Cancel Interaction Micro-Pattern: 2-Step Click (LOCKED)

**Pattern:** The cancel button starts as a controlled, restrained ghost button. One click arms it. A second click within 4 seconds fires the cancel. If not confirmed within 4 seconds, it disarms automatically.

**Rationale:** An undo toast means the cancel fires immediately (potentially allowing an SMS to slip through during the undo window if the scheduler fires within those 3 seconds). A modal is too heavy for a single-row action and interrupts Marco's scanning. The 2-step click keeps the confirmation inline at the row level, is reversible by doing nothing, and is fast to confirm intentionally. This pattern is common in destructive CLI tools and aligns with the "industrial precision" aesthetic.

**Step 1 (armed):**
- Button label changes from "CANCEL" to "CONFIRM?" (JetBrains Mono 12px, weight 700, uppercase)
- Button background changes from `--red-dim` to `--red` (full red)
- A thin red progress bar appears below the button, animating from full width to zero over 4 seconds (CSS `animation: countdown 4s linear forwards`)
- Job row background shifts to a very subtle red tint (`rgba(204, 34, 0, 0.06)`)
- Focus moves to the cancel button (see Accessibility §10) so that Enter or Space immediately fires Step 2

**Step 2 (confirmed):**
- DELETE request sent to server
- Row immediately collapses: `max-height` animates from measured height to 0, `opacity` to 0, `overflow: hidden`. Duration: 220ms. Easing: `cubic-bezier(0.4, 0, 1, 1)` (ease-in — quick exit).
- If the customer group now has no remaining rows, the group block also collapses with the same animation, delayed 220ms.
- The pending count in the header decrements immediately (optimistic update).

**Cancel fails (server error):**
- Row stays visible.
- Button label becomes "FAILED — RETRY?" in white on `--red` background.
- A one-line error message appears below the row in 14px JetBrains Mono `--text` (#E8E4DC): "Could not cancel. Server error — try again."
- Button disarms after 6 seconds and returns to normal "CANCEL" state.

**Disarm (4-second timeout, user did nothing):**
- Button label returns to "CANCEL". Background returns to `--red-dim`. Red tint on row disappears. All transitions: 150ms.

### Auto-Refresh

**Mechanism: 30-second polling (LOCKED)**

No Server-Sent Events. SSE requires the Go HTTP server to stay open with a long-lived connection, adds complexity to the `earlscheib.exe --admin` binary, and is overkill for a once-a-day tool. A 30-second poll is sufficient — the scheduler fires due jobs every 30 seconds on the server side, so polling at the same interval keeps the UI in sync.

**Behavior:**
- `setInterval` every 30,000ms calls `GET /earlscheibconcord/queue`.
- If data changed (compare JSON by ID set), re-render the job list with a fade-in stagger (same as page-load animation).
- If data unchanged, no visual change.
- The "Updated Xs ago" caption in the header ticks every second via a separate 1-second interval.
- During a pending cancel (button in "armed" state), polling is paused to avoid a mid-action re-render overwriting the armed state.

## 7. Technical Contracts

### Framework Choice: Vanilla JS + HTML Templates (LOCKED)

No Alpine.js, no HTMX, no Svelte SSR. The SPA is a single HTML file embedded via `go:embed`. Vanilla JS is the correct choice because:
1. No npm, no build step, no node_modules — the HTML file is the artifact.
2. The UI has exactly two states (list and empty/error) and exactly one action (cancel). Alpine.js (15KB) is unjustified overhead for this surface area.
3. The executor writes one file: `cmd/earlscheib/admin.html`. Zero toolchain configuration.
4. Template rendering uses a `renderQueue(data)` function that builds DOM nodes via `document.createElement` (not innerHTML string concatenation — XSS-safe by construction).

### CSS Approach: Pure CSS with Custom Properties (LOCKED)

No Tailwind CDN, no CSS-in-JS. All styling is `<style>` block in the single HTML file. Custom properties declared on `:root` for all tokens. CSS is authored directly — no utility classes. This keeps the file self-contained and avoids loading a 100KB Tailwind CDN payload over a local HTTP connection.

### Icon Approach: Inline SVG only (LOCKED)

No icon library CDN. Two SVG icons total, written as inline `<svg>` elements directly in the HTML:
1. A message/SMS icon for the page header branding mark (a minimal speech bubble, 20×20px)
2. A times/X icon for the cancel button (16×16px, stroke only, `stroke-width: 1.5`)

No third-party icons. No Lucide CDN. Inline keeps the page fully self-contained with no external network requests at render time.

### Auth Pattern: Magic-Link URL-Fragment Token (LOCKED)

**Decision:** The Go launcher (`earlscheib.exe --admin`) generates a random 256-bit token at startup using `crypto/rand`. It starts the local HTTP server, binds it to `127.0.0.1:RANDOM_PORT`, and opens the default browser to `http://127.0.0.1:PORT/#token=XXXXXXXXX`.

The SPA reads the token from `window.location.hash` on load. Every API call to the server includes the token as a header (`X-Admin-Token`). The Go HTTP server validates this token on every request (constant-time comparison). The token exists only for the lifetime of the process — no persistence, no cookies, no login form.

**Rationale for this choice over alternatives:**
- **HTTP Basic:** Requires a username/password prompt which Marco has never seen configured and will find confusing. Stored credentials in browser password manager for a localhost tool is also wrong.
- **Server-set cookie from one-time setup call:** More complex — requires a setup endpoint, session management, and cookie handling in Go. Overkill.
- **URL fragment token:** The token never appears in server logs (fragments are not sent in HTTP requests). The server only receives it as a header. If Marco shares a screenshot, the URL bar shows the fragment — acceptable risk for a localhost-only tool. The token rotates every time `--admin` is invoked. This is the simplest secure pattern for a single-user local tool.

**Implementation details:**
- Token: 32 bytes from `crypto/rand` → hex encoded (64 chars)
- Server validates: `X-Admin-Token` header, constant-time `hmac.Equal` or `subtle.ConstantTimeCompare`
- 401 response for missing or wrong token: JSON `{"error": "unauthorized"}` — not a redirect (SPA handles it)
- SPA on receiving 401: shows error state with copy "Session expired. Relaunch `earlscheib.exe --admin` to reconnect."

### Server Endpoint Shape (CONFIRMED + EXTENDED)

```
GET  /earlscheibconcord/queue
     Header: X-Admin-Token: {token}
     Header: X-EMS-Signature: {HMAC-SHA256 of empty body, using CCC_SECRET}
     Response 200:
     {
       "queued": [
         {
           "id": 42,
           "customer": "John Smith",
           "phone": "+19254441234",
           "send_at": 1745320800,          // Unix timestamp
           "send_at_fmt": "Tue Apr 22 · 10:00 AM PT",  // pre-formatted by server
           "repair_ref": "TEST-EMS-WATCHER-V1",  // doc_id from jobs table
           "message_type": "24h",           // "24h" | "3day" | "review"
           "body_preview": "Hi John, this is Earl Scheib Auto Body..."
         }
       ],
       "count": 1
     }
     Response 401: {"error": "unauthorized"}

DELETE /earlscheibconcord/queue/{id}
     Header: X-Admin-Token: {token}
     Header: X-EMS-Signature: {HMAC-SHA256 of request body (empty), using CCC_SECRET}
     Response 200: {"deleted": true, "id": 42}
     Response 404: {"error": "not found"}
     Response 401: {"error": "unauthorized"}
```

**Notes on data mapping (from `jobs` table):**
- `customer` ← `name` column
- `phone` ← `phone` column
- `send_at` ← `send_at` column (Unix timestamp, seconds)
- `repair_ref` ← `doc_id` column
- `message_type` ← `job_type` column (`"24h"` | `"3day"` | `"review"`)
- `body_preview` ← server renders the message template (`MSG_24H`, `MSG_3DAY`, `MSG_REVIEW`) truncated to 120 chars
- Only rows where `sent = 0` are included

**Both endpoints are HMAC-authenticated with the existing `CCC_SECRET` (same as all other routes) AND validated against the admin token. Both gates must pass.**

### Poll Interval

30 seconds. Defined above. `setInterval(fetchQueue, 30000)`.

## 12. Accessibility

### Focus States

The page has exactly one interactive element per job row: the cancel button. Tab order is designed for keyboard-first cancellation.

**Focus ring specification:**

```css
.cancel-button:focus-visible {
  outline: 2px solid var(--red);
  outline-offset: 2px;
}
```

Use `:focus-visible` (not `:focus`) so the ring appears only for keyboard navigation, not after mouse clicks. The 2px `--red` ring is consistent with the button's armed-state color, making the focused element immediately recognizable without introducing a new color.

**Tab order:**

1. The sticky header is skipped entirely — no focusable elements in the header. The pending count and timestamp are display-only.
2. Tab moves through job rows top-to-bottom.
3. Within each row, focus lands directly on the cancel button — the only interactive element. The job type badge, message preview, time, and repair ref are not focusable.
4. No skip-link is needed: the header contains no interactive controls, so the first Tab keypress lands immediately on the first cancel button in the content area.

**Keyboard cancel flow:**

After Step 1 (arm): focus is explicitly set (via `element.focus()`) to the cancel button of the armed row. This means the user can immediately press Enter or Space to fire Step 2 without a second mouse click or repositioning. This makes the 2-step cancel fully keyboard-operable.

```
Tab → [Cancel button, Row 1]  →  Space/Enter: arm  →  focus stays  →  Space/Enter: confirm
Tab → [Cancel button, Row 2]  →  Space/Enter: arm  →  ...
```

**Disarm via keyboard:** Pressing Escape while a button is armed disarms it (same as the 4-second timeout). Button returns to rest state, focus remains on the button.
